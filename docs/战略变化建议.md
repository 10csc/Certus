## Certus 项目分析报告

### 一、项目核心理解

Certus 是一个 **C++ Qt 桌面 + Python 子进程** 架构的跨 AI 平台研究搜索系统。核心价值链：

> 用户输入问题 → Planner 规划 → Orchestrator 通过 Playwright CDP 操控浏览器中的 AI 对话平台（DeepSeek/Kimi/ChatGPT/Gemini）→ Extractor 提取回复 → Synthesizer 交叉验证整合 → 输出 Markdown 研究报告

C++ 端负责 GUI、进程管理、浏览器生命周期、SQLite 存储；Python 端负责搜索编排、平台交互、内容提取。两端通过 **4 字节大端长度前缀 JSON 帧协议** 通信。

### 二、项目痛点与缺陷分析

#### 1. 架构层面

**1.1 orchestrator.py 职责过重（858 行，项目最大 Python 文件）**

[orchestrator.py](file:///f:/CodeFile/Certus/agent/orchestrator.py) 同时承担了：搜索发送、轮询等待提取、自进化联动（FailureAnalyzer + StrategyAdapter）、整合验证（含本地 API 降级）、最终审阅（final_review）、素材文件管理。这违反了项目自身的"最小设计原则"，且使该模块成为变更热点——任何搜索流程改动都必须改这个文件。

**1.2 配置双源漂移仍未完全消除**

[战略变化建议](file:///f:/CodeFile/Certus/docs/战略变化建议.md) §1.4 已指出此问题，且 [common.py](file:///f:/CodeFile/Certus/agent/common.py) 的 `load_config()` 已实现 Protocol 模式用 runtime_config、CLI 模式用 config.json 的分叉。但 `_save_session()` 仍然写 config.json（第 438-459 行），`get_or_create_project()` 也会 `save_config()` 写文件（第 133-144 行）。这意味着在 Protocol 模式下，Python 仍可能偷偷写 config.json，与"SQLite 唯一权威源"的设计矛盾。

**1.3 agent.py 单文件承载过多（585 行）**

[agent.py](file:///f:/CodeFile/Certus/agent/agent.py) 包含：主搜索流程（agent_search）、6 种 Mock 搜索函数、三线程模型（stdin_listener + heartbeat）、Protocol 模式入口、CLI 模式入口。架构文档 §11.4 已提到"若超过 300 行可拆分"，现在已近 600 行。

**1.4 自进化系统处于"半废弃"状态**

[evolution.py](file:///f:/CodeFile/Certus/agent/evolution.py)（746 行）仍然被 [orchestrator.py](file:///f:/CodeFile/Certus/agent/orchestrator.py) 的 `_wait_one()` 和 `execute()` 中多处调用（`FailureAnalyzer.analyze()`、`StrategyAdapter.adapt()`、`load_or_create_profile/polling()`），但 issues-log §1.2 已明确说"自进化不可用，已移除"。实际上代码没移除，只是移除了 C++ 端的自动触发入口。Python 端仍在搜索过程中静默调用自进化逻辑，这些调用散落在 orchestrator 的 try/except 中被吞掉（如第 343-348 行、第 570-582 行），既不报错也无可见效果。

#### 2. 稳定性与可靠性

**2.1 DOM 提取策略脆弱**

[extractor.py](file:///f:/CodeFile/Certus/agent/extractor.py) 的 `dom_extract()` 仅用一个选择器（从平台脚本的 `EXTRACT_SEL` 取）定位最后一个 AI 回复。如果平台 UI 改版（class 名变化），选择器立即失效。虽然 `_base.py` 做了输入框查找的多策略兜底，但**提取端没有类似的多策略兜底**。战略文档 §2.2 提出的 PlatformTyping 系统正是解决此问题，但目前处于"📋 后续"状态。

**2.2 内容完成判定依赖启发式标记**

`_wait_one()` 判定 AI 回复完成的逻辑（第 255-374 行）基于两个信号：内容长度稳定 + 结尾标记 `[搜索主题：xxx]`。无标记时靠 `no_closing_stable >= 10 轮`（约 30s）兜底。这种启发式判定：
- 对不提供结尾标记的平台（如 ChatGPT）效率低
- 对 AI 回复中间有停顿（streaming 间歇）的情况可能误判完成
- 阈值靠 evolution.py 自适应调整，但如前所述这条链路实际不可靠

**2.3 错误传播链路不完整**

[agent_protocol.py](file:///f:/CodeFile/Certus/agent/agent_protocol.py) 已修复了 pipe_broken 问题（issues-log §2.1），但 `on_event()` 只是 `write_frame()` 的别名（第 72-78 行），写失败后仅设 pipe_broken 标记。orchestrator 的调用方（如 `execute()` 第 482-490 行）不检查 `pipe_broken`，会继续调用 `on_event` 直到搜索自然结束。在管道破裂后，所有后续事件推送都是无效调用。

**2.4 心跳机制存在边界问题**

[_heartbeat()](file:///f:/CodeFile/Certus/agent/agent.py)（第 401-421 行）在 `cancel_flag.is_set()` 时退出循环——但 cancel_flag 是搜索取消标记，不是 heartbeat 应该退出的信号。如果搜索被取消但进程还没退出，heartbeat 线程会提前停止响应 pong，C++ 可能在 35s 后误判为僵死。

#### 3. 代码质量

**3.1 大量 try/except pass 静默吞异常**

orchestrator.py 中至少 8 处 `try: ... except Exception: pass`（第 343-348、366-371、570-582、629、771-777 行等），这些位置的异常被完全吞掉，不留任何日志。issues-log §2.1 的教训"进程间通信的每一处失败都要显式传播"尚未全面贯彻。

**3.2 API Key 通过 stdin 明文传输**

架构文档 §14.2 承认这一点：C++ 解密后通过 stdin JSON 传明文给 Python。虽然在进程管道内传输相对安全，但 `config.json` 中的 `deepseek_key` 字段在某些路径下也可能被写入文件（如 `save_config()` 不过滤 key 字段）。

**3.3 C++ AgentManager 注释过时**

[agentmanager.h](file:///f:/CodeFile/Certus/agent/core/agentmanager.h) 第 11 行仍写"崩溃检测 + 进化触发（五阶段 LLM 驱动自修复）"，但进化链路已移除。第 7 行仍写"搜索用 agent.py，进化用 evolution_agent.py"。这些注释会误导后续开发者。

**3.4 common.py 中 `sys` 未导入**

[safe_page_text()](file:///f:/CodeFile/Certus/agent/common.py) 第 373 行使用 `sys.stderr`，但 `import sys` 不在文件顶部——它依赖隐式导入（通过其他模块 side effect），这在独立测试时会崩溃。

#### 4. 功能缺失

**4.1 环境自举系统未实现**

战略文档 §2.1 提出的 `EnvironmentBootstrap` 仍处于"📋 后续"。当前 `validateConfig()` 仅检查 3 项（CDP 端口、API Key、Agent 路径），不检查 Playwright 安装、Chromium 可用、平台登录态、会话链接有效性。用户首次使用时，这些问题要到搜索失败时才暴露。

**4.2 缓存系统未实现**

战略文档 §2.4 的 `CostCache` 未实现。相同问题重复搜索会完整走一遍平台流程，浪费 token 和时间。

**4.3 平台定型系统未实现**

战略文档 §2.2 的 `PlatformTyping` 未实现。平台 UI 改版后，用户只能等搜索失败，然后去 RepairPage 手动诊断，没有一键重新适配能力。

**4.4 文件上传能力不完整**

orchestrator.py 的 L3 整合阶段会尝试 `plat_mod.upload_file(page, mat_file)`（第 687 行），但不是所有平台脚本都实现了此方法。失败时仅调 `_evolve_upload()` 记录诊断，不提供实际修复。

#### 5. 测试与工程化

**5.1 测试覆盖不均**

`agent/tests/` 有 14 个测试文件，但 Mock CDP 测试仅覆盖 6 种场景（正常/超时/拒绝/提取失败/整合OK/整合空）。orchestrator 的 L3 流程、cancel 机制的真实路径测试依赖手动端到端验证。

**5.2 无 CI/CD 流水线**

没有 `.github/workflows`、`Jenkinsfile` 或任何 CI 配置。C++ 构建依赖本地 `build_certus.bat`，Python 测试需手动运行。

**5.3 版本管理松散**

[version.txt](file:///f:/CodeFile/Certus/version.txt) 仅一个 `0.2.0`，无 changelog。Python Agent 版本在 [_get_agent_version()](file:///f:/CodeFile/Certus/agent/agent_protocol.py) 中 fallback 为 `"0.1.0"`（第 229 行），与实际不同步。

### 四、总体评价

Certus 在架构设计上思路清晰——C++/Python 进程隔离、JSON 帧协议、SQLite 独占写入、Named Mutex 防并发——这些都是正确的工程决策。issues-log 和战略文档也显示出团队有很强的反思和迭代能力。

当前最大的结构性风险是 **orchestrator.py 作为"上帝文件"持续膨胀**，以及 **自进化系统"半废弃"状态下的隐性调用**。前者阻碍可维护性，后者制造不确定性。

建议近期（1-2 周）集中精力做 P0/P1 修复 + orchestrator 拆分，为后续的平台定型和缓存系统打好基础。战略文档的"减肥增壮"方向完全正确，关键是执行节奏——先把主链路做硬，再考虑增壮。
